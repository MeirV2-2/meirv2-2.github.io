Module identity without the module.

A Windows x64 loader that convinces the loader graph a brand-new private allocation is a second instance of a signed Microsoft DLL — without touching the original on disk or in memory. Defeats integrity scanners. Keeps module attribution for stack walks.

What it does

Pick any signed Microsoft DLL the host process plausibly uses. Allocate a fresh private region. Hand-build an LDR_DATA_TABLE_ENTRY that claims to be a second instance of that DLL — and splice it into PEB->Ldr plus LdrpModuleBaseAddressIndex alongside the real one. The real DLL stays byte-identical to disk. The phantom carries your payload.

Stack walks, EnumProcessModules, and RtlPcToFileHeader all attribute the payload's code to the alias DLL. No threads start in private memory. No image bytes are modified. No ETW image-load events fire for the new region.

Features

Pure private

NtAllocateVirtualMemory(MEM_PRIVATE) for the payload. No module stomping, no SEC_IMAGE backing, no sacrificial DLL to keep alive.

Loader-graph alias

Phantom LDR_DATA_TABLE_ENTRY spliced into PEB->Ldr and inserted into LdrpModuleBaseAddressIndex (the AVL).

W^X enforced

Strict per-page protections. RX for code, R for read-only, RW for data. Never PAGE_EXECUTE_READWRITE, not even transiently.

Threadpool dispatch

No CreateThread. Payload runs via a TP_WAIT callback from ntdll!TppWorkerThread — clean stack origin.

Real unwind

Registers payload pdata via RtlAddGrowableFunctionTable so RtlVirtualUnwind walks cleanly through your frames.

PIC + DLL

Loads either a manually-mapped DLL or raw position-independent shellcode. Bring-your-own pdata/xdata is supported for PIC.

XOR encoding

On-disk payload can be XOR-encoded with any even-length hex key. Loader decodes in memory before mapping.

Pure NT

All APIs resolved by PEB walk + manual export-directory parse. No LoadLibrary, no GetProcAddress.

Build

make          # MinGW-w64 g++ (default)
make msvc     # MSVC cl.exe (from a vcvars64 shell)
build.bat     # Windows-only fallback (auto-detects VS 2022)

Use

phantomalias.exe agent.dll
phantomalias.exe beacon.bin --pic
phantomalias.exe beacon.bin --pic --alias wininet.dll
phantomalias.exe beacon.bin --pic --pic-pdata p.bin --pic-xdata x.bin
phantomalias.exe agent.enc --xor-key DEADBEEFCAFEBABE

For PIC payloads with stack-walkable frames, extract .text, .pdata, and .xdata from a built PE: 

python extract_pic.py build/beacon.dll out/beacon

Detection trade-off

The technique pays one tell: a single MEM_PRIVATE region with PAGE_EXECUTE_READ. That signal alone is noisy — Citrix, Adobe Reader's broker, .NET's JIT, V8's stub code, and a dozen other legitimate frameworks produce the same artifact. Production EDRs correlate it with other indicators. Everything else (thread origin, image bytes, module enumeration, unwind, ETW) stays consistent with the alias DLL.
